APRESENTAÇÃO DO SISTEMA – AquaSense
Sistema de Monitoramento de Drenagem Urbana
============================================
Data: 22/06/2026


========================================
1. O QUE É O SISTEMA
========================================

O AquaSense é um painel web que simula o monitoramento em tempo real de
sensores instalados em bueiros urbanos. Cada sensor mede três grandezas:

  – Obstrução (%)          → quão entupido está o bueiro
  – Precipitação (mm)      → volume de chuva no instante
  – Vazão (L/s)            → quanto líquido está passando pelo bueiro

O sistema gera leituras automaticamente a cada N segundos (configurável),
avalia os valores contra limiares de alerta e exibe tudo em tempo real
no navegador — sem necessidade de intervenção manual.


========================================
2. TECNOLOGIAS UTILIZADAS
========================================

BACKEND
-------
  Laravel 13.8     Framework PHP MVC. Gerencia rotas, controllers, models,
                   autenticação, migrations e seeders.

  PHP 8.5          Linguagem de programação. Usada em toda a lógica de
                   backend: geração de leituras, avaliação de alertas,
                   queries ao banco, renderização de views.

  MySQL 8.4        Banco de dados relacional. Armazena todas as tabelas
                   do sistema (sensores, leituras, alertas, etc.).
                   Executado via Laragon localmente.

  Eloquent ORM     Camada de abstração do Laravel para queries ao MySQL.
                   Cada tabela tem um Model PHP correspondente.

FRONTEND
--------
  Blade            Motor de templates do Laravel. Gera o HTML final
                   no servidor antes de enviar ao navegador.

  CSS puro         Interface estilizada sem frameworks (sem Bootstrap,
                   sem Tailwind). Usa variáveis CSS (--ink, --flow, etc.)
                   para manter consistência visual.

  JavaScript puro  Sem Vue, React ou jQuery. Apenas vanilla JS para:
                     – Relógio em tempo real
                     – Atualização dos cards de métricas via fetch()
                     – Timer global de auto-refresh (via localStorage)
                     – Mapa interativo

  MapLibre GL JS   Biblioteca de mapas vetoriais (open source). Usada na
                   tela do mapa operacional para exibir sensores como
                   marcadores coloridos por status.

AMBIENTE
--------
  Laragon          Servidor local (Apache + MySQL + PHP) para Windows.
                   O projeto roda em http://project.com (configurado
                   como virtual host no Laragon).

========================================
4. COMO AS PARTES SE CONECTAM
========================================

FLUXO DE UMA PÁGINA NORMAL
---------------------------

  Navegador
     │
     │  GET /dashboard (com cookie de sessão)
     ▼
  Laravel (web.php)
     │  verifica middleware 'auth' → usuário logado?
     ▼
  DashboardController@index
     │  busca sensores + última leitura de cada um (Eloquent)
     │  busca alertas ativos
     │  calcula médias (obstrução, chuva, vazão)
     ▼
  resources/views/dashboard/index.blade.php
     │  renderiza HTML com os dados PHP embutidos
     ▼
  Navegador exibe a página


FLUXO DO AUTO-REFRESH (JS)
--------------------------

Ao carregar qualquer página (exceto /map), o app.js inicia:

  1. Lê localStorage["aquasense_next_refresh"]
     → sabe exatamente quando a próxima atualização ocorre,
       independente de qual página o usuário navegou.

  2. Countdown tick a cada 1 segundo no rodapé ("Próx. atualização: 28s")

  3. Quando o timer chega a zero:
     a. Faz POST /api/leituras/gerar?force=1
        → ReadingGeneratorController chama SimulationService
        → SimulationService gera leitura para cada sensor ativo
        → Avalia alertas e cria/atualiza/resolve automaticamente
     b. Após resposta (ou timeout de 8s), recarrega a página (location.reload)
        → A página recarregada exibe os dados novos do banco

  4. Paralelamente, a cada 30 segundos, o JS faz GET /api/sensors
     → Atualiza os cards de métricas e os dots da sidebar sem recarregar

FLUXO DE GERAÇÃO DE LEITURAS (SimulationService)
-------------------------------------------------

  Para cada sensor ativo:

  1. Lê o modo de simulação da tabela configuracoes
     (sem_chuva / normal / chuva_fraca / chuva_forte / tempestade)

  2. Calcula precipitacao_mm
     → Depende do modo e de um fator aleatório com variação controlada

  3. Calcula obstrucao_pct
     → Random walk sobre a última leitura do sensor
     → Em modos de chuva, a tendência de subida é maior
     → Clamped entre 0% e 100%

  4. Calcula vazao_lps
     → fluxo_entrada = 10 + precipitacao * 11.6    (chuva = entrada d'água)
     → capacidade    = 300 * (1 - obstrucao * 0.98) (obstrução limita saída)
     → vazao         = min(fluxo_entrada, capacidade) + ruído aleatório

  5. Grava a leitura em leituras (INSERT)

  6. Avalia alertas (avaliarAlertas):
     → Determina severidade: critico ≥ 70%, risco ≥ 40%, atencao ≥ 10%
     → Existe alerta ativo para este sensor?
       SIM e severidade mudou → UPDATE no alerta existente
       SIM e severidade igual → nada
       NÃO e obstrucao OK     → nada
       NÃO e obstrucao > limiar → INSERT novo alerta
     → Se obstrucao voltou ao OK → resolve todos alertas ativos do sensor


========================================
5. BANCO DE DADOS – RELAÇÕES
========================================

  bairros (1) ──< sensores (N)   Um bairro tem vários sensores
  enderecos (1) ──< sensores (N) Um endereço pode ter vários sensores
  sensores (1) ──< leituras (N)  Um sensor gera muitas leituras
  sensores (1) ──< alertas (N)   Um sensor pode ter vários alertas

  Todos os relacionamentos têm ON DELETE CASCADE:
  → Excluir um bairro remove seus sensores, leituras e alertas
  → Excluir um sensor remove suas leituras e alertas

  Tipos de chave primária: INT UNSIGNED AUTO_INCREMENT (não BIGINT)
  Tipos de chave estrangeira: INT UNSIGNED


========================================
6. SISTEMA DE CONFIGURAÇÕES
========================================

As configurações ficam na tabela `configuracoes` (chave → valor).
Não há arquivo de config separado; tudo é lido do banco em tempo real.

  CHAVE                    VALOR PADRÃO   DESCRIÇÃO
  ─────────────────────────────────────────────────────────────────────
  intervalo_leitura_seg    60             Segundos entre coletas de dados
  limite_atencao           10             % obstrução que dispara Atenção
  limite_risco             40             % obstrução que dispara Risco
  limite_critico           70             % obstrução que dispara Crítico
  modo_simulacao           normal         Padrão climático da simulação
  modo_atualizacao         manual         manual ou automatico
  intervalo_atualizacao_seg 60            Segundos entre recarregamentos

Como o sistema lê:
  Setting::get('chave', valor_padrão)
    → Faz SELECT em configuracoes WHERE chave = ?
    → Se não encontrar, retorna o padrão passado como argumento

Como o JS recebe as configurações:
  O Blade emite as configs como tags <meta> no <head> de cada página:
    <meta name="refresh-mode"     content="{{ $modo }}">
    <meta name="refresh-interval" content="{{ $intervalo }}">
  O app.js lê essas metas com document.querySelector('meta[name="..."]').

MODOS DE SIMULAÇÃO
------------------
  sem_chuva    Precipitação mínima (0–0.30 mm). Obstrução tende a cair.
  normal       Simula período de chuva entre 14h–18h. Resto do dia: seco.
  chuva_fraca  Precipitação entre 1–5 mm. Pequena tendência de subida.
  chuva_forte  Precipitação entre 8–17 mm. Tendência clara de subida.
  tempestade   Precipitação entre 14–26 mm. Obstrução sobe rapidamente.

Alterar o modo em Configurações → efeito imediato na próxima coleta.

MODOS DE ATUALIZAÇÃO
--------------------
  manual       O botão "Atualizar" na barra superior gera uma leitura e
               recarrega a página. O JS não interfere.
  automatico   O JS controla um timer global (localStorage) e recarrega
               todas as páginas no mesmo ciclo, independente de navegação.

========================================
8. LOG DE COMANDOS SQL
========================================

  O AppServiceProvider registra um listener no evento DB::listen que
  captura todo INSERT, UPDATE e DELETE executado na aplicação.

  Cada registro salvo em log_consultas contém:
    – sql_query   : o SQL com placeholders (?)
    – bindings    : os valores reais em JSON
    – tempo_ms    : tempo de execução em milissegundos
    – executado_em: timestamp do comando

  O log é limitado aos últimos 200 registros automaticamente.
  Visível em: /comandos


========================================
9. COMO INICIAR DO ZERO
========================================

  Pré-requisitos: Laragon instalado, banco MySQL rodando.

  1. Clonar / copiar o projeto para C:\laragon\www\project\
  2. Copiar .env.example para .env e ajustar DB_DATABASE, DB_USERNAME etc.
  3. php artisan key:generate
  4. php artisan migrate:fresh
        → Cria todas as tabelas e insere as configurações padrão
  5. php artisan db:seed
        → Insere bairros, endereços e sensores (AquaSenseSeeder)
  6. Acessar http://project.com
  7. Login: admin@aquasense.com / password
  8. O sistema já começa a gerar leituras automaticamente ao acessar
     qualquer página (o JS dispara POST /api/leituras/gerar a cada ciclo)
